## Bugs Found & Fixed

### Bug 1 — Route stop dots always gray
**Where:** Routes page → Daily Runs → stop list and map pins
**What happened:** Every stop showed a gray dot regardless of status (complete, pending, etc.)
**Root cause:** JavaScript was checking `s.status === 'completed'` but the database stores `'complete'` (without the "d"). They never matched.
**Fix:** Changed all three occurrences to `'complete'`. Also updated the map pin color lookup table to use correct DB status keys (`complete`, `en_route`, `pending`).
**Status:** ✅ Fixed

---

### Bug 2 — Reports donut chart showing raw database codes
**Where:** Reports page → Order Status donut chart
**What happened:** Some slices in the donut chart were labeled with raw database codes instead of readable names (e.g., `pending_pickup` instead of "Pending Pickup").
**Root cause:** The chart's label lookup table was missing some legacy order statuses that still exist in the database.
**Fix:** Added `pending_pickup → "Pending Pickup"` and `ready → "Ready"` to both the labels and colors maps.
**Status:** ✅ Fixed

---

### Bug 3 — Driver stats showing "Unknown" row in Reports
**Where:** Reports page → Driver Performance table
**What happened:** The PM Delivery Run (assigned to Sofia Rodriguez as delivery driver) appeared as "Unknown" in the driver breakdown.
**Root cause:** The stats function only looked at `pickup_driver_id` on each route, completely ignoring `delivery_driver_id`. Delivery-only routes had no pickup driver, so they were skipped.
**Fix:** The function now credits both the pickup driver AND the delivery driver per route, attributing stops and hours to each.
**Status:** ✅ Fixed

---

### Bug 4 — All dates showing one day early
**Where:** Everywhere dates appear (orders list, customer panel, route cards, reports)
**What happened:** Dates were consistently off by one day — a route dated February 26 would show as February 25.
**Root cause:** Date-only strings like `'2026-02-26'` stored in the database are parsed by JavaScript as UTC midnight. In Pacific time (UTC-8), that's 4:00 PM the day *before*, so the date displayed as the previous day.
**Fix:** The `fmtDate()` function now detects date-only strings (exactly 10 characters) and appends `T00:00:00` before parsing, which forces JavaScript to treat it as local midnight.
**Status:** ✅ Fixed

---

### Bug 5 — Overview "Today's Routes" sorted alphabetically
**Where:** Overview page → "Today's Routes" card
**What happened:** Routes were listed A→Z by name instead of in time order. "AM Pickup Run" happened to be first alphabetically, but in general this would show routes out of order.
**Root cause:** The database query used `.order('name')`.
**Fix:** Changed to `.order('started_at')` with nulls last, so completed routes show first (in the order they ran), followed by in-progress, then scheduled routes (which have no start time yet). Name is kept as a secondary sort for ties.
**Status:** ✅ Fixed

---

### Bug 6 — Customer "Last Order" always showing a dash
**Where:** Customers list → "Last Order" column; Customer panel → stats strip
**What happened:** Every customer showed "—" for Last Order, even customers with multiple orders on file.
**Root cause:** The `last_order_at` field in the customers table had never been populated — orders were created but the field was never updated.
**Fix (two-part):**
1. **Database:** A SQL migration was run to backfill `last_order_at` for all existing customers based on their actual order history.
2. **Code:** The customer panel now computes `last_order_at` live from loaded orders as a real-time fallback, and writes it back to the database if it's missing — so new customers will self-heal the first time their panel is opened.
**Status:** ✅ Fixed

## Infrastructure Fixes Applied

**Missing RLS Policies on `route_stops`:** The route stops table had Row-Level Security enabled but no admin read policy — only driver-specific policies that referenced a deprecated `driver_id` column. This caused the "No stops on this run yet" blank state even with data present. Five permissive policies were added to allow admin and authenticated reads.

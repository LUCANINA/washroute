🗺️ How the Apps Connect

All three WashRoute apps share a single Supabase database. There is no direct app-to-app communication — they all read and write the same tables. The database is the source of truth for everything.

⚠️ Orphaned & Duplicate Fields

These columns exist in the database but are either never written, superseded by newer fields, or duplicated across tables.

routes — Legacy driver column

routes.driver_id is a leftover from before routes had separate pickup and delivery drivers. It's never written by any app today. The active fields are pickup_driver_id and delivery_driver_id.

route_stops — Duplicate photo fields

Two columns do the same thing: photo_url (old) and proof_photo_url (new). The driver app writes to proof_photo_url only. photo_url is written by admin in some flows. These should be unified.

customers — Fields never written

  • address_cache — JSONB column, never populated by any app
  • ambassador_code — never set by any app
  • total_orders — written by admin only; customer app doesn't increment it when placing orders

drivers — Live location never tracked

drivers.current_lat and drivers.current_lng exist for real-time location tracking. The driver app never writes to them. If live tracking was planned, it's not implemented.

orders — Duplicate address tracking

Customer address stored in multiple places: orders.pickup_address, orders.delivery_address, and separately in customers.address. They can drift out of sync if a customer updates their address.

services — Embedded legacy preferences

services.preferences (JSONB with short IDs like ch0z4vu) was the old preference system. The real system is the global preferences table (UUID IDs). The embedded field is now only used for price add-ons — but keeping both is confusing.

🔄 Cross-App Data Flow

How data moves between apps through the shared database:

Customer → Admin
Customer places order → writes to orders table → admin sees it in realtime (subscribed to orders changes) → admin assigns to a route → writes to routes and route_stops.
Admin → Driver
Admin assigns route → driver sees route_stops filtered by their driver_id → driver completes stops, uploads photos → writes route_stops.status, proof_photo_url, signature_data.
Driver → Customer
When driver updates route_stops.status to on-the-way, edge function notify-on-my-way fires → sends SMS to customer. Customer sees order status updated in their app when they refresh.
Admin → Customer
Edge function send-order-notification sends SMS/email on order status changes. Admin can also send direct messages via sms_messages/email_messages.
Customer ↔ Admin
Customer updates preferences → saved to customers.preferences (UUID format) → admin can read on customer profile. Admin can also override preferences directly.

🚨 Shared Write Risks

Tables that multiple apps write to can conflict if not carefully managed:

orders.status — 3-way writes

Admin, driver, and customer all update order status. There's no locking or conflict check. If an admin and driver update the same order simultaneously, one write silently wins. Low risk today (small volume), but worth monitoring as you scale.

route_stops — Admin + driver writes

Admin builds and edits stops; driver marks them complete and uploads photos. If admin edits a stop while a driver is actively working it, data could be overwritten. No version checking in place.

customers — Admin + customer writes

Admin can edit any customer field; customer can edit their own profile and preferences. Low risk since they're generally editing different fields, but admin overwrites of customer data will not notify the customer.

driver_messages — Safe two-way chat

Both admin and driver write to this table, but they're always inserting new rows (not updating), so there's no conflict risk here. ✓

✅ Cleanup Recommendations

Ordered from easiest/highest value to more involved. None of these are urgent, but they'll reduce confusion as the platform grows.

1
messages table — dropped ✓ Removed March 2026. Was superseded by sms_messages, email_messages, and driver_messages. The remaining 4 tables (notifications, order_items, daily_stats, subscriptions) are intentionally kept — they'll be wired up as those features are built.
2
Deprecate routes.driver_id The column routes.driver_id is never written. The real columns are pickup_driver_id and delivery_driver_id. If it's truly unused, it can be removed safely.
3
Unify route_stops photo field Decide on one field — recommend proof_photo_url — and update admin to write there consistently. Then photo_url can be retired.
4
Fix customers.total_orders counter Currently only admin increments this, not the customer app. Either remove it (it can be derived by counting orders) or make the customer app update it on order submission.
5
Implement driver location tracking (if desired) The DB has drivers.current_lat and drivers.current_lng ready to go. The driver app just needs to write GPS coordinates. This would enable live map tracking on the admin side.
6
Clarify services.preferences (embedded) vs. preferences table (global) Long-term, consider removing the embedded services.preferences JSONB entirely and moving any price-modifier logic to the global preferences table. This eliminates the dual-system confusion that caused the recent preference sync bug.
